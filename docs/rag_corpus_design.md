# RAG 语料与评测集设计说明

## 1. 目标

本项目的 RAG 第 0 阶段不是追求大规模知识库，而是先准备一套可运行、可验证、可扩展的中文工艺语料和评测集，支撑后续以下能力开发：

- 文档接入
- 文本切分
- 向量化与索引构建
- 检索与问答
- 来源引用
- 基础评测

本阶段强调“像真实工艺资料”，而不是“像教材”或“像百科”。因此语料会混合多种文体，并故意保留术语差异、信息分散和一定程度的表达冗余，以更接近现场资料库。

## 2. 语料设计原则

### 2.1 与现有系统的关系

现有系统的核心是结构化诊断和工艺优化，不依赖大模型排序。RAG 语料应作为并行新增能力的基础资产，而不是替代已有规则/案例引擎。

### 2.2 语料来源形态模拟

首批样本使用 Markdown 文档，模拟以下常见资料形态：

- 工艺操作/异常处理手册
- 对照表与 FAQ
- 问题复盘报告
- 班组长现场 SOP
- 工艺工程师周报或总结

这样设计的原因是，真实知识库通常不是单一格式，RAG 的切分、检索和引用能力需要在多种文体下都能工作。

### 2.3 内容覆盖原则

语料围绕当前项目已有工艺路线，优先覆盖以下高频问题：

- 虚焊、焊盘氧化、焊接不牢
- 导电浆料印刷缺陷、电阻偏高
- 固化不足、过烤、附着力下降
- 曝光对位偏移
- 贴合分层、起泡
- 模压后翘曲与回弹
- 耐压不良、绝缘失效

### 2.4 检索友好但不过度模板化

样本文档需要满足两个相反但都重要的目标：

- 便于后续切分、检索和引用
- 不要过度模板化，避免评测时仅靠标题命中

因此语料采用以下策略：

- 同一问题在不同文档中使用不同表达，例如“虚焊 / 假焊 / 焊接不实”
- 同一原因从不同角度描述，例如设备、材料、参数、操作动作
- 一些问题的完整答案分散在多个文档中
- 一些文档只有现象和原因，没有完整处理动作
- 一些文档是案例复盘，包含过程叙述和效果验证

## 4. 评测集设计原则

评测集不追求学术基准，而是服务于工程验收。首批评测集建议关注以下四类问题：

### 4.1 单文档直接检索

验证最基础的召回和回答能力，例如：

- 焊线后虚焊优先检查哪些因素
- 曝光对位偏移先看哪些设备项

### 4.2 跨文档拼接回答

验证是否能够将多个检索片段综合成一个回答，例如：

- 如何区分焊盘氧化与压力不足导致的虚焊
- 印刷后电阻偏高且固化波动时应如何联合判断

### 4.3 同义表达与现场口语

验证是否能处理非标准问法，例如：

- 假焊一般先看什么
- 贴合后鼓包怎么排查

### 4.4 拒答或知识不足

验证系统在资料不足时是否能克制回答，例如：

- 某新型激光工艺的最佳脉宽
- 某特定客户材料的私有配方比例

## 5. 评测字段约定

评测集中的每条样本建议包含以下字段：

- `id`：样本编号
- `question`：测试问题
- `expected_sources`：期望命中的文档名列表
- `expected_keywords`：答案中希望覆盖的关键词
- `answerable`：知识库中是否应能支撑回答
- `notes`：评测说明或人工关注点

## 6. 为什么先用 Markdown

首批语料以 Markdown 为主，主要出于以下考虑：

- 结构清晰，适合做基于标题和段落的切分
- 易于人工检查 chunk 边界
- 避免在 RAG 骨架尚未稳定时，把 PDF 文本抽取噪声与检索问题混在一起

后续可在 RAG MVP 稳定后再补 PDF，用于验证兼容性和鲁棒性。

## 7. 与后续代码实现的衔接

完成本阶段后，后续 RAG 开发应按以下顺序推进：

1. 定义 `Document`、`Chunk`、`Citation` 等数据结构
2. 实现文档读取和切分
3. 实现向量化和本地索引
4. 实现检索问答与来源引用
5. 运行评测集进行基础验收

## 8. 验收建议

第 0 阶段的验收重点不是“答案对不对”，而是“语料是否足以支撑开发和验证”。建议验收项包括：

- 文档主题覆盖当前核心工艺场景
- 文档之间存在可用于跨文档检索的互补信息
- 存在同义表达和非模板化叙述
- 评测集同时包含可回答与不可回答问题
- 文档和评测集命名清晰，便于后续程序读取与引用
